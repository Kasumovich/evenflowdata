# Deploying the Beige Book screen

A static site (`site/`) plus a data file (`site/data.js`) the pipeline regenerates.
Nothing here needs a running server — only a static host and an access layer for
email permissioning.

## Repo layout

```
site/index.html        the dashboard (presentation only)
site/data.js           data the dashboard reads (regenerated by the pipeline)
build_data.py          writes site/data.js (real pipeline, or --demo)
beigebook_altdata/      the scoring + forecast package
.github/workflows/update.yml   manual + scheduled refresh
requirements.txt
```

## 1. Put it on the web (Cloudflare Pages)

1. Push this repo to GitHub.
2. Cloudflare dashboard → **Workers & Pages → Create → Pages → Connect to Git**.
3. Build settings: **no build command**, **output directory = `site`**. Deploy.
   You now have a public URL like `your-screen.pages.dev`.

(Netlify or Vercel work the same way — point them at `site/` as the publish directory.)

## 2. Lock it to specific emails (Cloudflare Access)

This is where external sharing + permissioning happens. It is enforced at the edge,
**not in the page** — the page itself has no login and holds no secrets.

1. Cloudflare → **Zero Trust → Access → Applications → Add a self-hosted application**.
2. Application domain = your Pages URL.
3. Add a policy: **Action = Allow**, **Include → Emails** = the list of people you want.
   (Or "Emails ending in @yourfirm.com".)
4. Save. Visitors now get a one-time PIN to their email; only listed addresses get in.

To share with someone new, add their email to the policy. To revoke, remove it. No redeploy.

## 3. Updating the data

**On your command:** GitHub → **Actions → Update dashboard data → Run workflow**.
It regenerates `site/data.js`, commits, and Cloudflare Pages auto-rebuilds within a minute.

**Automatic on Beige Book release:** the same workflow runs daily on a cron. Once the
pipeline is wired to the live Fed archive (below), it processes a book only when a new one
has appeared and otherwise commits nothing — so it effectively fires ~8×/year on release.

The shipped workflow uses `--demo` so it works the moment you deploy. Switch to the real
pipeline by un-commenting the real `build_data.py` line in `update.yml`.

## 4. Going from demo to real data

`build_data.py --demo` ships synthetic series so everything renders immediately. For real
data, two seams in `beigebook_altdata/` need wiring (both flagged in code):

- `ingest.release_index()` → scrape the Fed Beige Book archive (1996+) and a FRASER map
  (pre-1996) for exact release dates.
- CPI label → set `FRED_API_KEY` (GitHub repo → Settings → Secrets) and switch
  `forecast.stub_cpi_yoy` to `forecast.fetch_cpi_yoy`.

The Beige Book and the Daily Treasury Statement need no API key.

## 5. CSV download

Built into the page — the **Download CSV** button exports the full lens history plus the
withholding overlay, named by the data's as-of date. No setup.

## What runs where

- The **browser** only renders `index.html` + `data.js`. No data pulls, no secrets.
- The **GitHub Action** does the live Fed/Treasury/FRED pulls and the model fits, then
  writes `data.js`. Secrets live in GitHub, never in the page.
- **Cloudflare Access** decides who can open the site. Email list lives there.
